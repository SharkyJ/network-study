XDP_GEN Deep Structural Analysis

Reverse-engineered internal architecture of the XDP_GEN eBPF hook from the eTran system, based on paper descriptions and Linux kernel XDP infrastructure.

What is XDP_GEN?

XDP_GEN is a new eBPF hook introduced by eTran that enables eBPF programs to generate packets inside the kernel — specifically ACK and credit packets for transport protocols. Standard XDP only processes ingress packets; XDP_GEN fills the gap for kernel-side packet generation without involving user-space.

Problem Addressed

Issue ②: No Packet Generation in XDP

Transports need to generate ACK/credit packets to coordinate senders and receivers, but the standard XDP hook has no mechanism for creating new packets inside the kernel.

Placement

Inside xdp_do_flush()

XDP_GEN is placed in the xdp_do_flush() (renamed dev_xdp_flush() in the paper) function, called at the end of each NAPI poll cycle.

Context

Same as XDP: struct xdp_md

Programs attached to XDP_GEN use the same packet context as standard XDP, enabling reuse of existing eBPF helpers and infrastructure.

Return Actions

XDP_TX / XDP_DROP / XDP_ABORTED

XDP_TX transmits the packet. XDP_DROP discards it. XDP_ABORTED terminates the generation loop — no more packets will be produced this cycle.

Key Design Decisions (from Paper §3.2.1)

Rather than dynamically allocating packet frames on-demand (which is slow due to kernel memory management), eTran uses batched pre-allocation via the page_pool allocator. The XDP_GEN program runs in a finite loop bounded by the NAPI budget. Each iteration may consume one pre-allocated frame to generate a packet.

Kernel-Side Hook Implementation (Reconstructed)

Based on the paper's description that XDP_GEN is placed in xdp_do_flush() and uses pre-allocated xdp_frame pools with a NAPI-budget-bounded loop, the following is the reconstructed kernel implementation structure.

1. Hook Placement: Inside xdp_do_flush()

In the vanilla Linux v6.6 kernel, xdp_do_flush() (in net/core/filter.c) flushes pending XDP redirect operations. eTran extends this function to call XDP_GEN after the flush.

/* net/core/filter.c (eTran-modified) — Reconstructed from paper §3.2.1 */ void xdp_do_flush(void) { /* Original flush logic: flush devmap, cpumap, xskmap bulk queues */ __dev_flush(); __cpu_map_flush(); __xsk_map_flush(); /* ★ eTran addition: Run XDP_GEN programs after ingress flush */ dev_xdp_gen_run(); /* New function added by eTran */ }

2. Core XDP_GEN Execution Loop

The paper states XDP_GEN runs in a "finite loop whose budget is the same as NAPI" and uses "pre-allocated frames from page_pool". The loop terminates on XDP_ABORTED or budget exhaustion.

/* ★ Reconstructed from paper §3.2.1 — dev_xdp_gen_run() */ static void dev_xdp_gen_run(void) { struct bpf_prog *prog = rcu_dereference(xdp_gen_prog); struct xdp_frame *frames; /* per-CPU pre-allocated pool */ struct xdp_buff xdp_buf; int budget = NAPI_POLL_WEIGHT; /* typically 64 */ int i, act; if (!prog) return; /* Pre-allocate frames from page_pool (amortized batch) */ frames = page_pool_alloc_batch(per_cpu_pool, budget); for (i = 0; i < budget; i++) { /* Initialize xdp_buff from pre-allocated frame */ xdp_init_buff_from_frame(&xdp_buf, &frames[i]); /* Run the eBPF program attached to XDP_GEN */ act = bpf_prog_run_xdp(prog, &xdp_buf); switch (act) { case XDP_TX: /* Transmit the generated ACK/credit packet */ dev_xdp_enqueue(&xdp_buf); /* wrapped as kfunc */ break; case XDP_DROP: /* Discard this frame, recycle to page_pool */ page_pool_recycle(&frames[i]); break; case XDP_ABORTED: /* No more packets to generate — exit loop */ page_pool_recycle_remaining(frames, i, budget); goto flush; } } flush: /* Flush all generated packets to NIC */ dev_xdp_flush(); }

3. Verifier Registration

Per paper §5: "eTran does not change the verifier but just registers new eBPF hooks and kfuncs." XDP_GEN reuses the XDP program type with an additional attach type.

/* kernel/bpf/verifier.c — Registration (reconstructed from paper §5) */ /* XDP_GEN uses the same context access checks as XDP */ /* The verifier checks: * (a) Program behaviors: bounded loops, NULL checks, no deadlock * (b) Context access: same as XDP (struct xdp_md) * (c) Helper/kfunc usage: SUBSET of XDP helpers * - EXCLUDES: bpf_redirect_map (prevent arbitrary redirect) * - EXCLUDES: bpf_sk_lookup_tcp (no socket access) * - INCLUDES: bpf_map_push/pop_elem, bpf_map_lookup_elem */

XDP_GEN Execution Lifecycle

This section traces the complete lifecycle of XDP_GEN execution during a single NAPI poll cycle, from packet arrival to ACK/credit transmission.

Sequence: TCP ACK Generation

01
NAPI poll begins. NIC driver (mlx5) receives incoming data packets. The XDP eBPF program is invoked for each packet in the driver's RX loop.
02
XDP program processes data packet. Looks up TCP connection state in eBPF hashmap using 4-tuple. Verifies ack/seq values. If valid, updates TCP window state.
03
XDP pushes ACK metadata. If an ACK needs to be sent, XDP calls bpf_map_push_elem() to push metadata (seq, ack number, connection ID) into a per-CPU eBPF queue.
04
XDP redirects data to AF_XDP. The data packet is redirected to user-space via AF_XDP socket (XDP_REDIRECT). Continues processing more packets until budget exhausted.
05
xdp_do_flush() is called. End of NAPI poll. Original flush logic runs (devmap, cpumap, xskmap). Then eTran's XDP_GEN hook is triggered.
06
XDP_GEN loop begins. Pre-allocated xdp_frame pool from page_pool is available. Loop iterates up to NAPI budget (64).
07
XDP_GEN pops metadata. Calls bpf_map_pop_elem() on the per-CPU queue. If empty → returns XDP_ABORTED to terminate loop.
08
XDP_GEN fills packet frame. Constructs ACK packet headers (Ethernet, IP, TCP) in the pre-allocated xdp_frame using popped metadata. Returns XDP_TX.
09
Kernel transmits the ACK. The XDP_TX action sends the generated ACK packet via dev_xdp_enqueue() to the NIC transmit queue.
10
Loop repeats or terminates. Steps 07–09 repeat for each queued ACK. On XDP_ABORTED or budget exhaustion, remaining frames are recycled and dev_xdp_flush() completes.
Key Performance Insight: ACK coalescing happens naturally — multiple data packets processed in one NAPI poll produce multiple ACK metadata entries, which XDP_GEN batches into ACK packets at the end. This mirrors coalescing in Google Falcon, TAS, Caladan, and eRPC.

Sequence: Homa Credit Generation

For Homa, the flow is similar but the XDP_GEN program is more complex:

H1
XDP_GEN selects RPCs from credit list. Uses bpf_rbtree_lower_bound() to find candidate RPCs ranked by priority (bytes remaining). Ensures max one RPC per peer.
H2
Looks up RPC state. Queries eBPF hashmap for RPC states, allocates credits based on remaining bytes and priority.
H3
Removes finished RPCs. Cleans up completed RPCs from the credit list (bpf_rbtree).
H4
Sends credit packets. Fills pre-allocated frames with credit packet data, returns XDP_TX. Uses bpf_tail_call to split complex logic across multiple programs to satisfy verifier instruction limits.

Data Structures Detail

per-CPU ACK Metadata Queue

The bridge between XDP (ingress) and XDP_GEN (packet generation). XDP pushes, XDP_GEN pops.

/* eBPF program side — per-CPU queue for ACK metadata */ /* This is a standard eBPF map, not a new type */ struct { __uint(type, BPF_MAP_TYPE_QUEUE); /* or PERCPU variant */ __uint(max_entries, 256); __type(value, struct ack_meta); } ack_queue SEC(".maps"); /* TCP ACK metadata structure (★ transport-specific) */ struct ack_meta { __u32 saddr, daddr; __u16 sport, dport; __u32 seq; __u32 ack_seq; __u16 window; /* Additional fields as needed by the transport */ }; /* In XDP program (ingress): */ bpf_map_push_elem(&ack_queue, &meta, BPF_EXIST); /* In XDP_GEN program (packet generation): */ int ret = bpf_map_pop_elem(&ack_queue, &meta); if (ret < 0) return XDP_ABORTED; /* Queue empty, stop generating */

PKT_QUEUE (BPF_MAP_TYPE_PKT_QUEUE) ★

New eBPF map type for pacing. Contains buckets with slots storing xdp_frame pointers.

/* ★ New eBPF map type added by eTran (kernel/bpf/) */ /* Reconstructed from paper §3.2.2 */ struct pkt_queue_bucket { struct xdp_frame *slots[BUCKET_SIZE]; int head, tail; /* For timing wheel: maps to a time slot */ /* For credit-based: maps to an RPC ID */ }; struct bpf_map_pkt_queue { struct bpf_map map; /* base eBPF map */ struct pkt_queue_bucket *buckets; __u32 nr_buckets; /* Timing wheel state (for rate-based pacing) */ __u64 current_tick; __u64 tick_interval_ns; }; /* Usage depends on pacing mechanism: */ /* * Rate-based (TCP/DCTCP): bucket_idx = tx_timestamp / tick_interval * → Implements Carousel-style timing wheel * * Credit-based (Homa): bucket_idx = RPC_ID * → Packets queued per-RPC, dequeued on credit replenishment */

eBPF Program Side (Transport Developer View)

This shows what a transport developer writes when implementing the XDP_GEN program. Based on the TCP and Homa implementations described in §4.

TCP ACK Generator (XDP_GEN program)

SEC("xdp_gen") int tcp_ack_gen(struct xdp_md *ctx) { struct ack_meta meta; int ret; /* Step 1: Pop ACK metadata from per-CPU queue */ ret = bpf_map_pop_elem(&ack_queue, &meta); if (ret < 0) return XDP_ABORTED; /* No more ACKs to send */ /* Step 2: Access the pre-allocated packet frame */ void *data = (void *)(long)ctx->data; void *data_end = (void *)(long)ctx->data_end; /* Step 3: Build Ethernet header */ struct ethhdr *eth = data; if ((void *)(eth + 1) > data_end) return XDP_DROP; /* Fill MAC addresses via bpf_fib_lookup or cached result */ /* Step 4: Build IP header */ struct iphdr *ip = (void *)(eth + 1); if ((void *)(ip + 1) > data_end) return XDP_DROP; ip->saddr = meta.saddr; ip->daddr = meta.daddr; ip->protocol = IPPROTO_TCP; ip->tot_len = bpf_htons(sizeof(*ip) + sizeof(struct tcphdr)); /* ... fill remaining IP fields ... */ /* Step 5: Build TCP ACK header */ struct tcphdr *tcp = (void *)(ip + 1); if ((void *)(tcp + 1) > data_end) return XDP_DROP; tcp->source = meta.sport; tcp->dest = meta.dport; tcp->seq = bpf_htonl(meta.seq); tcp->ack_seq = bpf_htonl(meta.ack_seq); tcp->ack = 1; tcp->window = bpf_htons(meta.window); /* ... compute checksum ... */ /* Step 6: Transmit the ACK packet */ return XDP_TX; }

Homa Credit Generator (XDP_GEN program, uses tail calls)

/* Homa uses bpf_tail_call to split complex logic (paper §4.2) */ /* States are passed via per-CPU variables in .bss section */ struct { __uint(type, BPF_MAP_TYPE_PROG_ARRAY); __uint(max_entries, 4); __type(key, __u32); __type(value, __u32); } gen_progs SEC(".maps"); /* Per-CPU state for passing between tail-called programs */ struct gen_state { __u64 last_min_bytes; __u32 last_peerid; __u32 credits_allocated; } gen_ctx SEC(".bss"); SEC("xdp_gen") int homa_credit_gen_phase1(struct xdp_md *ctx) { /* Phase 1: Select candidate RPCs from bpf_rbtree credit list */ /* Uses bpf_rbtree_lower_bound() to find RPCs by priority */ /* Ensures max 1 RPC per peer (emulates 2-level rbtree) */ /* Store selection in gen_ctx (.bss per-CPU) */ bpf_tail_call(ctx, &gen_progs, 1); /* → phase 2 */ return XDP_ABORTED; } SEC("xdp_gen/phase2") int homa_credit_gen_phase2(struct xdp_md *ctx) { /* Phase 2: Look up RPC state, allocate credits */ /* Phase 3: Remove finished RPCs from credit list */ bpf_tail_call(ctx, &gen_progs, 2); /* → phase 3 */ return XDP_ABORTED; } SEC("xdp_gen/phase3") int homa_credit_gen_phase3(struct xdp_md *ctx) { /* Phase 4: Build and send credit packets for selected RPCs */ /* Fill Ethernet/IP/Homa headers */ /* Return XDP_TX to transmit, XDP_ABORTED when done */ return XDP_TX; }
