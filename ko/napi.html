Linux 커널 네트워크 스택 심층 분석

NAPI와 NAPI Poll 주기
— 패킷 수신의 심장부

Linux 커널이 초당 수백만 패킷을 효율적으로 처리하는 핵심 메커니즘인 NAPI(New API)의 동작 원리를 상세히 설명하고, eTran/XDP_GEN이 이 메커니즘을 어떻게 활용하는지 분석합니다.

1. NAPI란 무엇인가?

NAPI(New API)는 Linux 커널 2.6(2003년)에 도입된 네트워크 패킷 수신 처리 프레임워크입니다. 기존의 순수 인터럽트 기반 수신 방식의 문제를 해결하기 위해, 인터럽트(interrupt)와 폴링(polling)을 혼합한 하이브리드 방식을 사용합니다.

🔔
비유: 초인종과 택배 수거

택배가 올 때마다 초인종을 누르는 방식(순수 인터럽트)을 생각해 보세요. 택배가 하루에 2~3개면 괜찮지만, 초당 수만 개의 택배가 쏟아지면 초인종 소리 처리만으로 하루가 끝납니다. NAPI는 이 문제를 다음과 같이 해결합니다:

① 첫 번째 택배가 오면 초인종이 울립니다 (인터럽트).
② 문을 열고 "이제부터 초인종 끄세요, 제가 직접 나가서 가져올게요"라고 합니다 (인터럽트 비활성화).
③ 현관 앞에 쌓인 택배를 한꺼번에 가져옵니다 — 최대 64개까지 (폴링, budget).
④ 더 이상 택배가 없으면 "다시 초인종 켜 주세요"라고 합니다 (인터럽트 재활성화).

이렇게 하면 택배가 많을 때는 효율적으로 배치 처리하고, 택배가 없을 때는 다른 일을 할 수 있습니다.

1.1 NAPI가 해결하는 문제

❌ NAPI 이전: 순수 인터럽트 방식

패킷마다 인터럽트 발생

NIC에 패킷이 도착할 때마다 하드웨어 인터럽트가 발생합니다. CPU는 현재 하던 작업을 중단하고 인터럽트 핸들러를 실행해야 합니다. 고속 네트워크에서는 초당 수백만 번의 인터럽트로 인해 CPU가 인터럽트 처리에만 매몰되는 "livelock" 상태에 빠집니다. 실제 패킷 처리보다 인터럽트 진입/복귀 오버헤드가 더 큽니다.

✅ NAPI: 인터럽트 + 폴링 혼합

첫 패킷만 인터럽트, 나머지는 폴링

첫 번째 패킷의 인터럽트로 "새 패킷이 왔구나"를 인지한 후, 인터럽트를 끄고 폴링(polling) 모드로 전환합니다. 이후 패킷은 CPU가 주도적으로 NIC 링 버퍼에서 꺼내옵니다. 이로써 인터럽트 오버헤드를 제거하고, 배치 처리로 효율을 높입니다.

핵심 아이디어: NAPI는 패킷 부하에 따라 적응적(adaptive)으로 동작합니다. 패킷이 적으면 인터럽트 기반(저지연), 패킷이 많으면 폴링 기반(고처리량). 두 세계의 장점을 결합한 것입니다.

2. NAPI의 핵심 구조

2.1 struct napi_struct

NAPI의 핵심 자료구조입니다. NIC 드라이버가 각 수신 큐(RX queue)마다 하나씩 생성합니다.

/* include/linux/netdevice.h (Linux v6.6 기준, 간략화) */ struct napi_struct { struct list_head poll_list; /* softirq의 poll 대기 리스트에 연결 */ unsigned long state; /* NAPI_STATE_SCHED, _DISABLE 등 */ int weight; /* poll 1회당 처리할 최대 패킷 수 (budget) */ /* 기본값: NAPI_POLL_WEIGHT = 64 */ int (*poll)(struct napi_struct *, int); /* ★ NIC 드라이버가 등록하는 poll 콜백 함수 */ /* 이 함수 안에서 실제 패킷 수신이 일어남 */ struct net_device *dev; /* 소속 네트워크 장치 (예: eth0) */ int rx_count; /* 이번 poll에서 처리한 패킷 수 */ struct gro_list gro_hash[]; /* GRO (Generic Receive Offload) 리스트 */ };

2.2 관련 핵심 상수

상수/설정기본값의미
NAPI_POLL_WEIGHT 64 poll 함수 1회 호출당 처리할 최대 패킷 수. 이 값이 "budget"
netdev_budget 300 softirq 1회 실행당 전체 NAPI 인스턴스를 합산하여 처리할 최대 패킷 수

NIC Ring Buffer 크기 256~4096 NIC 하드웨어의 수신 디스크립터 링 크기. 드라이버가 여기서 패킷을 가져옴

3. NAPI Poll 주기 — 한 사이클의 전체 흐름

하나의 NAPI poll 주기는 인터럽트부터 flush까지의 완전한 사이클입니다. 아래에서 각 단계를 상세히 추적합니다.

NAPI Poll 주기 전체 흐름도

3.1 각 단계 상세 설명

①
NIC 하드웨어 인터럽트 발생. 패킷이 NIC에 도착하면, NIC은 DMA를 통해 패킷을 Ring Buffer(수신 디스크립터 링)에 기록하고, CPU에 하드웨어 인터럽트(IRQ)를 발생시킵니다.
②
인터럽트 핸들러 실행. NIC 드라이버의 인터럽트 핸들러가 실행됩니다. 이 핸들러는 아주 짧고 간단합니다 — NIC 인터럽트를 비활성화하고 napi_schedule()을 호출하여 softirq를 예약합니다. 패킷을 직접 처리하지 않습니다.
③
NET_RX_SOFTIRQ 트리거. napi_schedule()이 napi_struct를 per-CPU poll 리스트에 추가하고, NET_RX_SOFTIRQ (소프트 인터럽트)를 예약합니다. softirq는 하드웨어 인터럽트보다 낮은 우선순위에서 실행되며, 실제 패킷 처리를 담당합니다.
④
napi->poll(budget) 호출. softirq 핸들러(net_rx_action)가 poll 리스트에서 napi_struct를 꺼내어 드라이버가 등록한 poll 콜백 함수를 호출합니다. 인자로 budget = 64를 전달합니다. 이것이 "NAPI poll" 입니다.
⑤
Ring Buffer에서 패킷 가져오기 루프. 드라이버의 poll 함수 내부에서 for 루프를 돌며 NIC의 Ring Buffer에서 패킷을 하나씩 가져옵니다. budget (기본 64)개까지 처리합니다. 각 패킷에 대해 sk_buff를 할당하거나, XDP가 활성화된 경우 먼저 XDP를 실행합니다.
⑥
각 패킷에 대한 처리. XDP hook이 활성화되어 있으면 먼저 XDP eBPF 프로그램을 실행합니다. XDP_PASS면 GRO(Generic Receive Offload)를 거쳐 커널 스택으로 전달되고, XDP_REDIRECT면 AF_XDP 등으로 리다이렉트되며, XDP_DROP이면 폐기됩니다.
⑦
XDP 판정에 따른 분기. eTran 환경에서는 대부분의 데이터 패킷이 XDP_REDIRECT로 AF_XDP를 통해 user-space로 전달되고, ACK 메타데이터는 per-CPU 큐에 push됩니다.
⑧
xdp_do_flush() 호출. poll 루프가 끝나면 (또는 budget 소진 시) XDP redirect 등의 보류 작업을 flush합니다. devmap, cpumap, xskmap의 벌크 큐를 flush합니다.
⑨
dev_xdp_gen_run() — ★ eTran이 추가한 단계. xdp_do_flush() 내부에서 XDP_GEN hook이 실행됩니다. 사전 할당된 xdp_frame에 ACK/Credit 패킷을 채우고 NIC으로 전송합니다. 이것이 XDP_GEN이 NAPI poll 주기에 삽입되는 정확한 지점입니다.
⑩
napi_complete_done() — poll 완료. 처리한 패킷 수(work)가 budget보다 적으면 "더 이상 패킷이 없다"고 판단하여 NIC 인터럽트를 재활성화하고 IDLE 상태로 돌아갑니다. 만약 work == budget이면 아직 패킷이 남아 있을 수 있으므로 poll 리스트에 유지되어 다음 softirq에서 다시 poll됩니다.

4. Budget — Poll 루프의 제한 장치

Budget은 NAPI poll이 한 번에 처리할 수 있는 최대 패킷 수입니다. 이 제한이 있는 이유는 네트워크가 CPU를 독점하지 못하게 하기 위함입니다.

⏳
비유: 뷔페 접시 제한

뷔페에서 한 번에 접시 64개까지만 들고 올 수 있다는 규칙(budget=64)이 있습니다. 64개를 다 가져왔는데 아직 음식이 남아 있으면 줄 끝으로 가서 다시 기다립니다(다음 poll 주기). 64개를 채우기 전에 음식이 떨어지면 "다 가져왔어요"라고 말합니다(napi_complete). 이 규칙은 한 사람이 뷔페를 독점하는 것(CPU 독점)을 방지합니다.

/* net/core/dev.c — net_rx_action() 간략화 */ static void net_rx_action(struct softirq_action *h) { unsigned long time_limit = jiffies + usecs_to_jiffies(2000); /* 2ms */ int budget = netdev_budget; /* 기본 300 */ for (;;) { struct napi_struct *n = list_first_entry(&poll_list); /* 드라이버의 poll 콜백 호출 — 이것이 "NAPI poll" */ int work = n->poll(n, min(n->weight, budget)); budget -= work; /* 종료 조건: budget 소진 또는 시간 초과 */ if (budget <= 0 || time_after(jiffies, time_limit)) break; } }

4.1 Poll 완료 판단 로직

work < budget (예: 30 < 64)

✅ Poll 완료 (Complete)

Ring Buffer의 패킷을 모두 처리했음을 의미합니다. 드라이버가 napi_complete_done()을 호출하여 NIC 인터럽트를 재활성화합니다. NAPI는 IDLE 상태로 돌아갑니다. 다음 패킷이 오면 새로운 인터럽트로 다시 시작합니다.

work == budget (64 == 64)

🔄 아직 패킷 남음 (Reschedule)

Budget을 모두 사용했지만 Ring Buffer에 아직 패킷이 남아 있을 수 있습니다. napi_complete_done()을 호출하지 않습니다. NAPI는 poll 리스트에 유지되어 다음 softirq 실행에서 다시 poll됩니다. 인터럽트는 비활성화 상태를 유지합니다.

2단계 budget 시스템: NAPI에는 두 가지 budget이 있습니다.
(1) per-NAPI budget = napi->weight (기본 64) — 하나의 NAPI 인스턴스가 poll 1회에 처리할 패킷 수.
(2) 전역 budget = netdev_budget (기본 300) — softirq 1회 실행에서 모든 NAPI 인스턴스가 합산하여 처리할 총 패킷 수.
또한 netdev_budget_usecs (기본 2ms)의 시간 제한도 있어, budget이 남아도 2ms가 지나면 softirq가 종료됩니다.

5. NIC Ring Buffer와 NAPI의 관계

NIC과 커널 사이의 패킷 전달은 Ring Buffer(순환 큐)를 통해 이루어집니다. NAPI poll은 이 Ring Buffer에서 패킷을 꺼내는 것입니다.

NIC Ring Buffer → NAPI poll → 커널 스택 데이터 흐름

5.1 멀티 큐와 RSS

현대 NIC(mlx5, i40e 등)은 여러 개의 RX 큐를 가지고 있으며, RSS(Receive Side Scaling)를 사용하여 패킷의 flow hash에 따라 큐를 분산시킵니다. 각 큐는 자체적인 napi_struct를 가지며, 서로 다른 CPU 코어에서 독립적으로 poll됩니다.

RX Queue 0

CPU 코어 0

napi_struct #0
독립적 poll 루프
독립적 budget

RX Queue 1

CPU 코어 1

napi_struct #1
독립적 poll 루프
독립적 budget

RX Queue N

CPU 코어 N

napi_struct #N
독립적 poll 루프
독립적 budget

핵심: 멀티 큐 + RSS 덕분에, NAPI poll은 코어별로 병렬 실행됩니다. 각 코어는 자기 큐에서만 패킷을 가져오므로 교차 코어 락이 필요 없습니다. eTran의 per-CPU 큐와 per-CPU xdp_frame 풀은 이 구조를 자연스럽게 활용합니다.

6. 고급 주제: Busy Polling과 적응적 인터럽트

6.1 Busy Polling

지연 시간에 민감한 애플리케이션을 위해, Linux는 Busy Polling 모드도 지원합니다. 애플리케이션이 소켓에서 recv()를 호출할 때, softirq를 기다리지 않고 애플리케이션 스레드가 직접 NIC poll을 수행합니다.

일반 NAPI

인터럽트 → softirq → poll

패킷 도착 → IRQ → softirq 스케줄링 → net_rx_action에서 poll. 지연 시간은 softirq 스케줄링 대기 시간 포함.

Busy Polling

앱 스레드가 직접 poll

recv() 호출 시 앱 스레드가 직접 napi->poll() 실행. softirq 대기 없이 최소 지연으로 패킷 수신. CPU를 더 사용하는 대신 지연 감소.

6.2 적응적 인터럽트 병합 (Adaptive Interrupt Coalescing)

현대 NIC은 인터럽트 병합(Interrupt Coalescing)을 지원합니다 — 패킷이 올 때마다 즉시 인터럽트를 발생시키는 대신, 일정 시간 또는 일정 패킷 수를 모아서 한 번에 인터럽트를 발생시킵니다.

설정의미효과
rx-usecs 인터럽트 발생 전 대기 시간 (마이크로초) ↑ = 인터럽트 수 감소, 처리량 증가, 지연 증가
rx-frames 인터럽트 발생 전 대기 패킷 수 ↑ = 배치 효과 증가, 지연 증가
adaptive-rx 부하에 따라 자동 조절 저부하 시 즉시 인터럽트(저지연), 고부하 시 병합(고처리량)
# NIC 인터럽트 병합 설정 확인/변경 $ ethtool -c eth0 # 현재 설정 확인 $ ethtool -C eth0 adaptive-rx on # 적응적 병합 활성화 $ ethtool -C eth0 rx-usecs 50 # 50μs 대기 후 인터럽트

7. GRO — NAPI Poll 내부의 패킷 병합 최적화

GRO(Generic Receive Offload)는 NAPI poll 루프 내에서 수행되는 최적화로, 같은 flow(동일 연결)에 속하는 작은 패킷들을 하나의 큰 패킷으로 병합합니다.

📦
비유: 택배 합포장

같은 사람에게 가는 작은 소포 10개를 각각 배송하는 대신, 하나의 큰 박스에 합쳐서 보내면 배송비(커널 스택 처리 비용)가 10분의 1로 줄어듭니다. GRO는 NAPI poll 단계에서 이 합포장을 수행합니다.

GRO와 NAPI의 관계: GRO는 NAPI poll 루프 내에서 napi_gro_receive() 호출을 통해 실행됩니다. 같은 flow의 패킷들이 하나의 NAPI poll에서 여러 개 처리될 때, 이들을 병합하여 커널 스택에는 하나의 큰 sk_buff로 전달합니다. 이로써 sk_buff 할당 횟수와 스택 순회 횟수가 줄어듭니다.

8. eTran이 NAPI를 활용하는 방식

eTran은 NAPI poll 주기의 구조를 깊이 이해하고, 그 안에 XDP_GEN을 자연스럽게 삽입합니다.

8.1 XDP_GEN의 정확한 삽입 지점

패킷 수신 루프 (기존 XDP)
Ring Buffer에서 패킷 가져오기 → XDP 프로그램 실행 → ACK 메타데이터 push / 데이터 AF_XDP 리다이렉트
bpf_map_push_elem(&ack_queue, &meta)
수신 루프 종료

8.2 왜 이 위치가 최적인가?

장점 1: 자연스러운 ACK 병합

배치 처리 효과

하나의 NAPI poll에서 64개의 데이터 패킷을 수신하면, 각각에 대한 ACK 메타데이터가 큐에 쌓입니다. XDP_GEN은 poll 끝에서 이를 한꺼번에 처리하므로, 마지막 cumulative ACK 하나만 보내도 64개를 모두 확인할 수 있습니다.

장점 2: 컨텍스트 스위치 없음

동일 softirq 컨텍스트

XDP_GEN은 NAPI softirq 컨텍스트 내에서 실행되므로, 수신 처리와 ACK 전송 사이에 커널-유저 전환이나 softirq 경계 전환이 없습니다. 같은 CPU에서 연속적으로 처리됩니다.

장점 3: per-CPU 격리

락 없는 처리

NAPI poll은 per-CPU이므로, XDP가 push한 메타데이터와 XDP_GEN이 pop하는 메타데이터가 같은 CPU의 per-CPU 큐에서 처리됩니다. 교차 코어 동기화가 전혀 필요 없습니다.

장점 4: 페이지 풀 재활용

할당 비용 제거

XDP_GEN은 NAPI 컨텍스트의 page_pool 할당자를 사용하여 프레임을 사전 할당합니다. 사용하지 않은 프레임은 풀로 돌아가고, 다음 poll에서 재활용됩니다. 매번 새로 할당할 필요가 없습니다.

9. 핵심 정리

개념설명
NAPI인터럽트 + 폴링 혼합 방식의 패킷 수신 프레임워크. 고속 네트워크에서 인터럽트 오버헤드를 제거
NAPI poll드라이버가 등록한 콜백 함수. Ring Buffer에서 패킷을 budget만큼 가져와 처리
Budgetpoll 1회에 처리할 최대 패킷 수 (기본 64). CPU 독점 방지
Poll 주기인터럽트 → softirq → poll 루프 → flush → complete의 전체 사이클
완료 조건work < budget면 완료 (인터럽트 재활성화). work == budget면 재스케줄
멀티 큐 + RSS여러 RX 큐를 여러 코어에 분산. 각 큐는 독립적 NAPI 인스턴스
GRONAPI poll 내에서 같은 flow의 패킷을 병합하여 스택 비용 감소
XDP_GEN 삽입 지점xdp_do_flush() 내부, poll 루프 직후. ACK 병합과 per-CPU 격리를 자연스럽게 활용
eTran과의 관계 한 줄 요약: NAPI poll 주기는 "패킷 수신 → 처리 → flush"의 자연스러운 배치 사이클을 제공하는데, eTran은 이 사이클의 flush 단계에 XDP_GEN을 삽입하여, 수신한 패킷에 대한 ACK/Credit 생성을 같은 CPU, 같은 softirq 컨텍스트에서, 컨텍스트 스위치 없이, 락 없이, 사전 할당된 프레임으로 배치 처리합니다.
