Linux 커널 네트워크 스택 분석

sk_buff 구조체 심층 분석

Linux 네트워크 스택의 핵심 데이터 구조인 sk_buff의 내부 구조를 파헤치고, 이것이 고속 네트워크에서 성능 병목이 되는 이유와 eTran/XDP_GEN이 이를 어떻게 우회하는지 설명합니다.

1. sk_buff란 무엇인가?

sk_buff(socket buffer)는 Linux 커널에서 네트워크 패킷 하나를 표현하는 메타데이터 구조체입니다. 패킷이 NIC에 도착한 순간부터 애플리케이션에 전달될 때까지, 커널 네트워크 스택의 모든 계층이 이 구조체를 통해 패킷을 다룹니다.

📦
비유: 택배 운송장

택배 상자(패킷 데이터)에 붙어 있는 운송장이 sk_buff입니다. 운송장에는 보내는 사람, 받는 사람, 분류 코드, 취급 주의 사항, 배송 추적 이력 등이 적혀 있습니다. 택배가 물류센터를 거칠 때마다 각 센터가 운송장에 도장을 찍고 정보를 추가합니다. 문제는 이 운송장이 상자보다 더 클 수도 있을 만큼 거대하다는 것입니다.

핵심 사실: sk_buff는 약 232바이트 크기의 구조체입니다. ACK 패킷(54바이트)을 하나 보내기 위해, 패킷 데이터보다 4배 이상 큰 메타데이터를 할당하고 초기화해야 합니다.

2. sk_buff 내부 구조

2.1 메모리 레이아웃

sk_buff는 패킷 데이터 자체가 아니라, 패킷에 대한 메타데이터를 담는 구조체입니다. 실제 패킷 데이터는 별도의 버퍼에 존재하며, sk_buff의 포인터들이 이 버퍼를 가리킵니다.

2.3 4개 포인터의 동작 원리

head/data/tail/end 포인터는 패킷이 커널 스택을 통과하면서 동적으로 이동합니다.

/* 패킷이 커널 스택을 올라가면서 data 포인터가 이동하는 과정 */ /* NIC 드라이버가 패킷을 받으면: */ skb->data = skb->head + headroom; /* Ethernet 헤더 시작 */ skb->tail = skb->data + pkt_len; /* L2 처리 후 → L3로 전달 (Ethernet 헤더를 "벗겨냄"): */ skb_pull(skb, ETH_HLEN); /* data 포인터를 14B 앞으로 이동 */ /* 이제 skb->data는 IP 헤더를 가리킴 */ /* L3 처리 후 → L4로 전달 (IP 헤더를 "벗겨냄"): */ skb_pull(skb, ip_hdr_len); /* data를 또 앞으로 이동 */ /* 이제 skb->data는 TCP 헤더를 가리킴 */ /* 송신 시에는 반대로 — 헤더를 "추가"할 때: */ skb_push(skb, sizeof(struct tcphdr)); /* data를 뒤로 이동 */ /* headroom에 새 TCP 헤더를 쓸 공간 확보 */
🧅
비유: 양파 껍질 벗기기

수신 시: 양파(패킷)의 껍질(헤더)을 바깥부터 하나씩 벗깁니다. Ethernet → IP → TCP 순서로 각 계층이 자기 헤더를 처리하고 data 포인터를 앞으로 밀어 다음 계층에게 넘깁니다.
송신 시: 반대로 알맹이(페이로드)에 껍질(헤더)을 하나씩 감쌉니다. TCP → IP → Ethernet 순서로 data를 뒤로 당겨 헤더 공간을 확보합니다.

3. sk_buff가 성능 병목이 되는 4가지 이유

저속 네트워크(1 Gbps 이하)에서는 sk_buff의 오버헤드가 문제되지 않았습니다. 하지만 10/25/100 Gbps 고속 네트워크에서 초당 수백만 패킷을 처리해야 할 때, 매 패킷마다 반복되는 이 오버헤드는 치명적인 병목이 됩니다.

병목 #1

매 패킷마다 무거운 메모리 할당/해제

패킷이 도착할 때마다 커널은 kmem_cache_alloc()을 호출하여 ~232바이트의 sk_buff와 별도의 패킷 데이터 버퍼를 할당합니다. 패킷 처리 후에는 kfree_skb()로 해제합니다.

25 Gbps 네트워크에서 64바이트 패킷 기준 초당 ~37 Mpps(3,700만 패킷)가 도착합니다. 매 패킷마다 할당/해제를 수행하면 slab 할당자의 freelist 관리, 캐시 라인 오염, NUMA 노드 간 메모리 접근 등이 누적됩니다.

✅ XDP_GEN 해결책: page_pool에서 xdp_frame을 배치(batch)로 사전 할당하고 재활용합니다. NAPI poll 시작 시 한 번에 64개 프레임을 확보하여, 루프 내에서는 할당 비용이 거의 0입니다.
병목 #2

구조체가 지나치게 크고 범용적

sk_buff는 TCP, UDP, ICMP, SCTP, IPsec, 터널링, netfilter, 트래픽 제어 등 커널의 모든 네트워크 기능을 지원해야 합니다. 그래서 대부분의 패킷에서 사용하지 않는 필드까지 포함합니다.

ACK 패킷 하나(54바이트)를 보내는 데 필요한 정보는 src/dst IP, src/dst 포트, seq, ack_seq, window 정도로 ~30바이트면 충분합니다. 하지만 232바이트 전체를 할당하고 초기화해야 합니다. 이는 CPU 캐시를 불필요하게 소모합니다.

✅ XDP_GEN 해결책: xdp_frame(~64바이트)과 전송 프로토콜에 특화된 ack_meta(~30바이트) 구조체만 사용합니다. 필요한 필드만 담은 최소한의 메타데이터로 패킷을 생성합니다.
병목 #3

전체 프로토콜 스택 순회 강제

sk_buff는 커널 네트워크 스택의 "공용 화물"이므로, ACK 하나를 보내려 해도 아래의 전체 경로를 통과해야 합니다:

/* TCP ACK 전송의 커널 경로 (sk_buff 사용) */ tcp_send_ack() → alloc_skb() /* ① sk_buff 할당 */ → tcp_transmit_skb() /* ② TCP 헤더 구성 */ → icsk_af_ops->queue_xmit() /* ③ IP 계층 전달 */ → ip_queue_xmit() /* ④ 라우팅 조회 */ → ip_local_out() /* ⑤ netfilter LOCAL_OUT */ → ip_output() /* ⑥ netfilter POST_ROUTING */ → dev_queue_xmit() /* ⑦ Qdisc 트래픽 제어 */ → dev_hard_start_xmit() /* ⑧ NIC 드라이버 전송 */

각 계층은 sk_buff의 필드를 읽고 쓰며, 도중에 netfilter hook, conntrack, Qdisc 등 부가 서브시스템도 개입합니다. 54바이트짜리 ACK에도 이 전체 경로가 적용됩니다.

✅ XDP_GEN 해결책: 이 전체 스택을 완전히 건너뜁니다. XDP_GEN 프로그램이 사전 할당된 프레임에 직접 헤더를 채우고, dev_xdp_enqueue() kfunc 한 번 호출로 NIC 드라이버에 직접 전송합니다. 중간 계층이 전혀 없습니다.
병목 #4

참조 카운트와 락에 의한 멀티코어 경합

sk_buff의 수명은 refcount_t users (atomic 참조 카운트)로 관리됩니다. 또한 소켓 수신 큐(sk->sk_receive_queue)에 패킷을 넣을 때 spin lock을 잡아야 합니다.

멀티코어 환경에서 atomic 연산과 spin lock은 캐시 라인 바운싱(cache line bouncing)을 유발합니다 — 하나의 코어가 캐시 라인을 수정하면, 그 라인을 가진 다른 모든 코어의 캐시가 무효화(invalidate)되어 다시 메모리에서 읽어와야 합니다.

✅ XDP_GEN 해결책: per-CPU 자료구조만 사용합니다. per-CPU 큐, per-CPU xdp_frame 풀 — 각 코어가 자기만의 데이터를 다루므로 교차 코어 동기화가 거의 없습니다. atomic 연산도, spin lock도 필요하지 않습니다.

6. 핵심 정리

한 줄 요약: sk_buff는 Linux 커널의 모든 네트워크 기능을 지원하기 위해 설계된 범용 패킷 메타데이터 구조체인데, 그 범용성 때문에 구조체가 크고, 할당이 무겁고, 전체 프로토콜 스택을 통과해야 하며, 멀티코어 동기화가 필요합니다. 고속 네트워크에서 ACK 같은 단순한 제어 패킷까지 이 무거운 경로를 거치게 하는 것이 병목이며, XDP_GEN은 sk_buff를 아예 사용하지 않고 최소한의 메타데이터와 사전 할당된 프레임으로 패킷을 직접 생성·전송하여 이 병목을 제거합니다.
sk_buff의 장점

범용성과 호환성

TCP, UDP, ICMP, IPsec, 터널링, 방화벽, QoS 등 커널의 모든 네트워크 기능이 하나의 통일된 패킷 표현으로 동작할 수 있습니다. 30년간 검증된 안정적인 추상화입니다.

XDP_GEN의 장점

특화된 고속 경로

ACK/Credit 같은 단순 패킷에 한정하여, 불필요한 범용 인프라를 모두 제거한 최소 경로를 제공합니다. 범용성을 포기한 대신 극한의 성능을 달성합니다.
