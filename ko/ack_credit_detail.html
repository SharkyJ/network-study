eTran / XDP_GEN 교육 자료

ACK 패킷과 Credit 패킷
— 쉬운 설명

XDP_GEN이 커널 내부에서 생성하는 두 종류의 패킷이 무엇이고, 왜 필요한지, 어떻게 동작하는지를 비유와 함께 설명합니다.

1. 먼저 — 왜 "응답 패킷"이 필요한가?

네트워크에서 데이터를 보내는 것은 일방통행 도로가 아닙니다. 보낸 쪽은 반드시 "받는 쪽이 잘 받았는지"를 알아야 합니다. 그렇지 않으면 데이터가 중간에 사라져도 아무도 모릅니다.

📬
비유: 등기 우편

일반 우편은 보내고 잊으면 끝이지만, 등기 우편은 수취인이 "받았습니다"라는 수취 확인서에 서명하고, 이것이 보낸 사람에게 돌아옵니다. 이 수취 확인서가 바로 네트워크에서의 ACK 패킷입니다. 그리고 수취인이 "다음 우편은 3통까지만 보내주세요"라고 요청서를 보내는 것이 Credit 패킷입니다.

전송 프로토콜은 크게 두 가지 종류의 "응답 패킷"을 사용합니다:

ACK 패킷

📨 "잘 받았어요"

수신자가 송신자에게 "당신이 보낸 데이터 패킷을 정상적으로 받았습니다"라고 알려주는 확인 패킷입니다. TCP 같은 송신자 주도(Sender-driven) 프로토콜에서 주로 사용됩니다.

Credit 패킷

🎟️ "이만큼 보내도 됩니다"

수신자가 송신자에게 "앞으로 이만큼의 데이터를 보내도 좋습니다"라는 허가증을 발급하는 패킷입니다. Homa 같은 수신자 주도(Receiver-driven) 프로토콜에서 사용됩니다.

2. ACK 패킷 — 상세 설명

2.1 ACK가 하는 일

TCP 통신에서 ACK(Acknowledgment) 패킷은 3가지 핵심 기능을 수행합니다:

기능설명비유
수신 확인 수신자가 "seq 번호 X까지의 데이터를 정상적으로 받았습니다"라고 송신자에게 알립니다. 📋 택배 수취 확인 서명
손실 감지 같은 ack 번호가 3번 연속 도착하면(3 Duplicate ACK), 송신자는 "그 패킷이 분실되었구나"라고 판단하고 재전송합니다. 🔁 "아직 안 왔어요"를 3번 반복하면 다시 보내기
흐름 제어 ACK에 포함된 window 값으로 "나는 현재 이만큼의 버퍼 여유가 있습니다"를 알려, 송신자가 과도하게 보내지 않도록 합니다. 🪣 "내 양동이에 빈 공간이 100L 남았어요"

2.2 ACK 패킷의 실제 구조

XDP_GEN이 생성하는 TCP ACK 패킷은 아래와 같은 계층 구조를 가집니다:

Ethernet 헤더 14 bytes src/dst MAC
IP 헤더 20 bytes src/dst IP
TCP 헤더 20 bytes seq, ack_seq, window, ACK 플래그
페이로드 0 bytes ACK는 데이터 없음
핵심 포인트: ACK 패킷은 데이터를 담지 않는 매우 작은 패킷입니다 (약 54 bytes). 하지만 이 작은 패킷이 없으면 TCP 통신 전체가 작동하지 않습니다. 이 작은 패킷의 생성 속도가 전체 전송 성능에 직접적인 영향을 미칩니다.

2.3 기존 방식 vs. XDP_GEN 방식

❌ 기존 Linux TCP 방식

느린 경로

데이터 패킷 수신 → sk_buff 할당 → TCP 스택 전체 통과 → 소켓 계층 → ACK 생성 → IP 계층 → 드라이버로 전송. 수많은 커널 함수와 락(lock)을 거치므로 느립니다.

✅ eTran XDP_GEN 방식

빠른 경로

데이터 패킷 수신(XDP) → 메타데이터만 큐에 push → NAPI 끝에서 XDP_GEN이 메타데이터 pop → 사전 할당된 프레임에 헤더 직접 구성 → NIC으로 바로 전송. sk_buff 없이 최소 경로로 처리됩니다.

2.4 XDP_GEN의 TCP ACK 생성 흐름

TCP ACK 생성 과정 (한 NAPI poll 주기 내)
ACK 병합(Coalescing) 효과: 하나의 NAPI poll에서 여러 데이터 패킷을 수신하면, 각각에 대한 ACK 메타데이터가 큐에 쌓입니다. XDP_GEN은 이를 한꺼번에 처리하므로, 개별 ACK를 보내는 것보다 효율적입니다. 예를 들어 10개의 데이터 패킷을 받았다면, 마지막 ack_seq 값 하나만 ACK로 보내도 10개 모두를 확인할 수 있습니다.

3. Credit 패킷 — 상세 설명

3.1 Credit이 하는 일

🎫
비유: 놀이공원 식권

놀이공원에서 밥을 먹으려면 먼저 식권(Credit)을 받아야 합니다. 식권 없이는 음식을 주문할 수 없습니다. 식당(수신자)이 현재 처리 가능한 양만큼만 식권을 발급하므로, 손님(송신자)이 한꺼번에 몰려 음식이 넘치는(네트워크 혼잡) 상황을 방지합니다. 또한 식당은 빨리 나갈 수 있는 간단한 메뉴(작은 RPC) 주문에 우선적으로 식권을 발급하여 대기 시간을 최소화합니다. 이것이 바로 Homa의 SRPT(Shortest Remaining Processing Time) 정책입니다.

Credit 패킷은 수신자 주도(Receiver-driven) 전송 프로토콜인 Homa에서 핵심적인 역할을 합니다. ACK와는 근본적으로 목적이 다릅니다:

구분ACK 패킷Credit 패킷
방향 수신자 → 송신자 수신자 → 송신자
의미 "보낸 데이터 받았어요"
(과거에 대한 확인)		 "데이터를 보내도 돼요"
(미래에 대한 허가)
목적 신뢰성 보장 (손실 감지, 재전송) 혼잡 제어 (네트워크 과부하 방지)
타이밍 데이터를 받은 후 데이터를 보내기 전
사용 프로토콜 TCP (DCTCP), 송신자 주도 Homa, NDP 등 수신자 주도
포함 정보 ack_seq, window RPC ID, 허용 바이트 수, 우선순위

3.2 Homa에서 Credit의 동작 방식

Homa 프로토콜은 RPC(Remote Procedure Call) 단위로 통신합니다. 하나의 RPC 메시지는 여러 패킷으로 나뉘어 전송되는데, 이 과정이 Credit으로 조절됩니다:

①
비예약(Unscheduled) 패킷 전송. 송신자는 Credit 없이도 RPC의 처음 몇 패킷을 보낼 수 있습니다 (비예약 패킷). 이것은 수신자가 해당 RPC의 존재를 알게 하기 위함입니다.
②
수신자가 Credit 발급. 비예약 패킷을 받은 수신자는 남은 데이터량과 다른 RPC들의 상태를 고려하여 Credit 패킷을 보냅니다. "이 RPC에 대해 N 바이트까지 더 보내도 됩니다."
③
SRPT 우선순위 적용. 여러 RPC가 경쟁할 때, 수신자는 남은 바이트가 가장 적은 RPC에 먼저 Credit을 발급합니다. 이렇게 하면 작은 RPC가 빨리 완료되어 평균 지연 시간이 줄어듭니다.
④
송신자가 Credit에 따라 전송. Credit을 받은 송신자는 허가된 양만큼만 데이터를 전송합니다. Credit이 없으면 패킷은 페이싱 엔진(PKT_QUEUE)에 대기합니다.

3.3 Credit 패킷의 구조

Ethernet 14 bytes
IP 헤더 20 bytes
Homa Credit 헤더 RPC ID, offset, priority "RPC #42에 대해 offset 8000까지 전송 허용"

3.4 XDP_GEN에서의 Credit 생성 흐름

Homa Credit 생성 과정 (XDP_GEN 내부)
왜 Credit 생성이 더 복잡한가? TCP ACK는 단순히 "받았다"는 확인이지만, Homa Credit은 스케줄링 결정을 내려야 합니다. "여러 RPC 중 어떤 것에 먼저 Credit을 줄 것인가?" — 이것이 bpf_rbtree를 사용한 SRPT 우선순위 결정이며, eBPF 코드가 5,349줄로 TCP(2,173줄)보다 2.5배 더 큰 이유입니다.

4. 왜 커널 내부(XDP_GEN)에서 생성해야 하는가?

ACK와 Credit 패킷을 user-space에서 생성할 수도 있습니다. 그런데 eTran은 왜 굳이 커널 내부에서 이를 처리할까요?

4.1 성능: 커널-유저 전환 비용 제거

❌
User-space 방식: 데이터 패킷 수신 → 커널→유저 전환 → 유저에서 ACK 생성 → 유저→커널 전환 → NIC 전송. 매 패킷마다 최소 2번의 컨텍스트 스위치가 발생합니다.
✅
XDP_GEN 방식: 데이터 패킷 수신(XDP) → 같은 NAPI 컨텍스트에서 ACK 생성(XDP_GEN) → NIC 전송. 컨텍스트 스위치 없이 커널 내에서 모든 것이 완료됩니다.

4.2 보호(Protection): 전송 상태 보호

ACK와 Credit은 전송 프로토콜의 핵심 제어 메커니즘입니다. 만약 악의적인 애플리케이션이 가짜 ACK를 보내거나 Credit을 조작하면, 전체 네트워크 통신이 망가질 수 있습니다.

위험 시나리오

가짜 ACK 공격

악의적 앱이 실제로 받지 않은 데이터에 대해 ACK를 보내면, 송신자는 데이터가 전달된 것으로 착각합니다. 결과적으로 데이터가 영구적으로 손실됩니다.

위험 시나리오

Credit 남용

악의적 앱이 자신에게 과도한 Credit을 할당하면, 네트워크 대역폭을 독점하여 다른 애플리케이션의 통신을 방해합니다.

XDP_GEN은 커널 eBPF 내부에서 실행되므로, user-space 애플리케이션이 전송 상태에 직접 접근하거나 ACK/Credit 생성을 조작할 수 없습니다. eBPF verifier가 프로그램의 안전성을 보장합니다.

4.3 지연 시간: 최소 경로 확보

⏱️
비유: 우체국 자동 응답 시스템

편지를 받을 때마다 사람이 직접 확인하고 답장을 쓰는 것(user-space)보다, 우체국 기계가 자동으로 수취 확인서를 발행하는 것(XDP_GEN)이 훨씬 빠릅니다. XDP_GEN은 ACK 생성을 NIC 드라이버의 패킷 수신 경로(NAPI)에서 직접 수행하므로, 데이터가 도착한 바로 그 시점에 최소 지연으로 ACK를 보낼 수 있습니다.

6. 한 줄 요약

🎯
정리

ACK 패킷은 "잘 받았으니 다음 걸 보내세요"라는 과거에 대한 확인이고,
Credit 패킷은 "이만큼 보내도 됩니다"라는 미래에 대한 허가입니다.

둘 다 전송 프로토콜의 핵심인데, 기존에는 커널 TCP 스택 전체를 거쳐야 해서 느렸습니다. XDP_GEN은 이 패킷들을 NIC 드라이버 수준에서 커널 eBPF 내부적으로 직접 생성·전송하여, 성능을 극대화하면서도 보안과 격리를 유지합니다.
