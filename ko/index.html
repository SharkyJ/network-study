NSDI '25 — eTran 논문 분석

XDP_GEN 심층 구조 분석

eTran 시스템의 XDP_GEN eBPF hook에 대한 내부 아키텍처를 논문 기술 내용과 Linux 커널 XDP 인프라를 기반으로 역공학 분석한 결과입니다.

해당 논문의 실제 구현 코드는 찾을 수 없었음. 비공개 상태.

XDP_GEN이란?

XDP_GEN은 eTran이 도입한 새로운 eBPF hook으로, eBPF 프로그램이 커널 내부에서 패킷을 생성할 수 있게 합니다 — 구체적으로 전송 프로토콜을 위한 ACK 및 credit 패킷을 생성합니다. 표준 XDP는 수신(ingress) 패킷만 처리하므로, XDP_GEN은 user-space 개입 없이 커널 측 패킷 생성이라는 공백을 채웁니다.

해결하는 문제

이슈 ②: XDP에서 패킷 생성 불가

전송 프로토콜은 송신자와 수신자를 조율하기 위해 ACK/credit 패킷을 생성해야 하지만, 표준 XDP hook에는 커널 내부에서 새로운 패킷을 생성하는 메커니즘이 없습니다.

배치 위치

xdp_do_flush() 내부

XDP_GEN은 각 NAPI poll 주기가 끝날 때 호출되는 xdp_do_flush()(논문에서는 dev_xdp_flush()로 지칭) 함수 내부에 배치됩니다.

컨텍스트

XDP와 동일: struct xdp_md

XDP_GEN에 연결된 프로그램은 표준 XDP와 동일한 패킷 컨텍스트를 사용하여, 기존 eBPF helper 및 인프라의 재사용이 가능합니다.

반환 액션

XDP_TX / XDP_DROP / XDP_ABORTED

XDP_TX는 패킷을 전송합니다. XDP_DROP은 폐기합니다. XDP_ABORTED는 생성 루프를 종료합니다 — 이번 주기에서 더 이상 패킷이 생성되지 않습니다.

핵심 설계 결정 (논문 §3.2.1)

필요 시 동적으로 패킷 프레임을 할당하는 방식(커널 메모리 관리의 복잡성으로 인해 느림) 대신, eTran은 page_pool 할당자를 통한 배치(batch) 사전 할당을 사용합니다. XDP_GEN 프로그램은 NAPI budget으로 제한된 유한 루프에서 실행됩니다. 각 반복(iteration)에서 사전 할당된 프레임 하나를 소비하여 패킷을 생성할 수 있습니다.

전통적인 TCP/IP 스택

sk_buff가 IP 및 TCP 계층을 통해 전파될 때 상태 머신은 시퀀스 번호를 평가하고 ACK가 필요한지 결정한다. 필요한 경우 커널은 새 sk_buff를 할당하고, 이더넷, IP, TCP 헤더를 채우며, 라우팅 메트릭을 위해 FIB (Forwarding Information Base)를 쿼리하고, 큐 규율(qdisc)을 통해 버퍼를 NIC 드라이버로 밀어 넣는다. 이러한 동기식 할당 및 라우팅 오버헤드는 커널 네트워크에서 지연 시간의 주요 원인이 된다.

pagepool 설명

Linux 커널의 `page_pool`은 네트워크 스택, 특히 XDP(eXpress Data Path)와 같은 고속 패킷 처리를 위해 최적화된 특수 메모리 할당자(Memory Allocator)입니다. 고속 네트워크 환경을 이해하기 위해 `page_pool`의 등장 배경과 핵심 원리를 상세하고 알기 쉽게 설명해 드리겠습니다. ### 1. 등장 배경 및 도입 목적 전통적인 Linux 네트워크 스택에서는 NIC가 패킷을 수신할 때마다 커널의 표준 메모리 할당자를 통해 메모리를 동적으로 할당하고, 처리가 끝나면 다시 시스템에 반환하는 과정을 거쳤습니다. 그러나 네트워크 속도가 기가비트 이상으로 빨라지면서 이러한 메모리 할당/해제 작업 자체가 막대한 CPU 오버헤드와 캐시 미스(Cache miss)를 유발하는 병목 지점이 되었습니다. `page_pool`은 이러한 한계를 극복하기 위해 다음과 같은 목표를 가지고 도입되었습니다: * **DMA 매핑 상태 유지:** 패킷이 담길 메모리 페이지가 풀(Pool) 내에 머무는 동안 하드웨어 장치(NIC)와 직접 데이터를 주고받을 수 있는 DMA(Direct Memory Access) 매핑 상태를 그대로 유지합니다. 이로 인해 무거운 DMA 동기화 및 매핑 해제/재설정 비용이 제거됩니다. * **초고속 재활용(Fast Recycling):** 커널의 범용 페이지 할당자를 거치지 않고, 네트워크 전용으로 구축된 캐시에서 메모리를 즉시 재활용하여 처리 속도를 극대화합니다. * **NIC 드라이버 구현의 표준화:** 파편화되어 있던 드라이버별 메모리 재활용 코드를 공통된 API로 제공하여 네트워크 드라이버의 구조를 단순화합니다. ### 2. 핵심 동작 원리 `page_pool`의 가장 큰 특징은 **Lockless(잠금 없는) 캐싱과 재활용 메커니즘**입니다. * **NAPI와 결합된 Lockless 설계:** `page_pool`은 원칙적으로 하드웨어 큐(Queue)의 수와 동일하게 생성됩니다. 이 풀은 이전에 설명해 드린 NAPI의 softirq(소프트웨어 인터럽트) 컨텍스트 내에서 동작하도록 보장되므로, 여러 스레드 간의 충돌을 방지하기 위한 Lock(잠금) 연산 없이 안전하고 빠르게 메모리를 할당할 수 있습니다. * **자체 캐시 (Fast Cache / Ring Cache):** NIC 드라이버가 패킷 수신을 위해 메모리를 요청하면, 범용 메모리 영역이 아니라 `page_pool`이 자체적으로 보유한 캐시(Fast cache 또는 ptr-ring)에서 메모리를 즉시 내어줍니다. * **스마트한 메모리 재활용:** 수신된 패킷이 네트워크 계층이나 XDP 프로그램을 거쳐 더 이상 필요 없어지면(예: 폐기되거나 전송이 완료된 경우), 시스템으로 반환되지 않습니다. `page_pool_put_page` 같은 API가 호출되어 참조 카운트(refcnt)를 확인한 후, 안전하다고 판단되면 즉시 `page_pool`의 캐시로 되돌려 보냅니다. 또한 페이지 분할(Page splitting) 없이 전체 페이지를 활용하기 때문에 메모리를 관리하는 구조체(`struct page`)의 캐시 라인 오염(Cache line dirtying)도 발생하지 않습니다. ### 3. eTran 및 XDP와의 관계 `page_pool`은 본래 한 페이지에 하나의 프레임을 할당하여 처리 속도를 극도로 끌어올리는 XDP 모드에 최적화되어 설계되었습니다. 앞선 보고서에서 분석한 eTran 아키텍처의 `XDP_GEN` 메커니즘이 바로 이 `page_pool`을 창의적으로 활용한 사례입니다. eTran은 패킷 수신(RX) 용도로 만들어진 `page_pool` 인프라를 송신(TX) 경로의 패킷 생성용으로 전환하여 사용합니다. NAPI 루프가 끝날 때 ACK 등 제어 패킷을 동적으로 생성하면 락 경합과 할당 지연이 발생하지만, eTran은 `page_pool`에서 미리 할당해 둔 프레임들을 가져와 순식간에 패킷을 일괄 생성함으로써 막대한 CPU 오버헤드 없이 고속 처리를 달성할 수 있었습니다. 요약하자면, `page_pool`은 네트워크 전용으로 **"한 번 빌려온 메모리를 커널 시스템에 반납하지 않고, NAPI라는 안전한 환경 안에서 NIC 드라이버 혼자 락(Lock) 없이 계속 돌려쓰는(재활용하는) 고속 메모리 창고"**라고 이해하시면 쉽습니다.

커널 측 Hook 구현 (재구성)

논문에서 XDP_GEN이 xdp_do_flush() 내부에 배치되고, 사전 할당된 xdp_frame 풀과 NAPI-budget 제한 루프를 사용한다는 기술을 바탕으로 재구성한 커널 구현 구조입니다.

1. Hook 배치 위치: xdp_do_flush() 내부

표준 Linux v6.6 커널에서 xdp_do_flush()(net/core/filter.c)는 보류 중인 XDP redirect 작업을 flush합니다. eTran은 이 함수를 확장하여 flush 후 XDP_GEN을 호출합니다.

/* net/core/filter.c (eTran 수정) — 논문 §3.2.1에서 재구성 */ void xdp_do_flush(void) { /* 기존 flush 로직: devmap, cpumap, xskmap 벌크 큐 flush */ __dev_flush(); __cpu_map_flush(); __xsk_map_flush(); /* ★ eTran 추가: 수신 flush 후 XDP_GEN 프로그램 실행 */ dev_xdp_gen_run(); /* eTran이 추가한 새 함수 */ }

2. XDP_GEN 핵심 실행 루프

논문에 따르면 XDP_GEN은 "NAPI와 동일한 budget의 유한 루프"에서 실행되며 "page_pool에서 사전 할당된 프레임"을 사용합니다. 루프는 XDP_ABORTED 또는 budget 소진 시 종료됩니다.

/* ★ 논문 §3.2.1에서 재구성 — dev_xdp_gen_run() */ static void dev_xdp_gen_run(void) { struct bpf_prog *prog = rcu_dereference(xdp_gen_prog); struct xdp_frame *frames; /* per-CPU 사전 할당 풀 */ struct xdp_buff xdp_buf; int budget = NAPI_POLL_WEIGHT; /* 일반적으로 64 */ int i, act; if (!prog) return; /* page_pool에서 프레임 사전 할당 (배치 분할 상각) */ frames = page_pool_alloc_batch(per_cpu_pool, budget); for (i = 0; i < budget; i++) { /* 사전 할당된 프레임으로 xdp_buff 초기화 */ xdp_init_buff_from_frame(&xdp_buf, &frames[i]); /* XDP_GEN에 연결된 eBPF 프로그램 실행 */ act = bpf_prog_run_xdp(prog, &xdp_buf); switch (act) { case XDP_TX: /* 생성된 ACK/credit 패킷 전송 */ dev_xdp_enqueue(&xdp_buf); /* kfunc으로 래핑됨 */ break; case XDP_DROP: /* 이 프레임 폐기, page_pool로 재활용 */ page_pool_recycle(&frames[i]); break; case XDP_ABORTED: /* 더 이상 생성할 패킷 없음 — 루프 종료 */ page_pool_recycle_remaining(frames, i, budget); goto flush; } } flush: /* 생성된 모든 패킷을 NIC으로 flush */ dev_xdp_flush(); }

3. Verifier 등록

논문 §5에 따르면: "eTran은 verifier를 변경하지 않고 새로운 eBPF hook과 kfunc만 verifier에 등록합니다." XDP_GEN은 추가 attach type과 함께 XDP 프로그램 타입을 재사용합니다.

XDP_GEN 실행 생명주기

이 섹션은 단일 NAPI poll 주기 동안 패킷 도착부터 ACK/credit 전송까지 XDP_GEN 실행의 전체 생명주기를 추적합니다.

시퀀스: TCP ACK 생성

01
NAPI poll 시작. NIC 드라이버(mlx5)가 수신 데이터 패킷을 받습니다. 드라이버의 RX 루프에서 각 패킷에 대해 XDP eBPF 프로그램이 호출됩니다.
02
XDP 프로그램이 데이터 패킷 처리. 4-tuple을 사용하여 eBPF hashmap에서 TCP 연결 상태를 조회합니다. ack/seq 값을 검증합니다. 유효하면 TCP 윈도우 상태를 갱신합니다.
03
XDP가 ACK 메타데이터를 push. ACK를 보내야 하는 경우, XDP는 bpf_map_push_elem()을 호출하여 메타데이터(seq, ack 번호, 연결 ID)를 per-CPU eBPF 큐에 push합니다.
04
XDP가 데이터를 AF_XDP로 리다이렉트. 데이터 패킷은 AF_XDP 소켓을 통해 user-space로 리다이렉트됩니다(XDP_REDIRECT). budget이 소진될 때까지 더 많은 패킷을 계속 처리합니다.
05
xdp_do_flush() 호출. NAPI poll 종료. 기존 flush 로직 실행(devmap, cpumap, xskmap). 그 후 eTran의 XDP_GEN hook이 트리거됩니다.
06
XDP_GEN 루프 시작. page_pool에서 사전 할당된 xdp_frame 풀이 사용 가능합니다. NAPI budget(64)까지 루프가 반복됩니다.
07
XDP_GEN이 메타데이터 pop. per-CPU 큐에서 bpf_map_pop_elem()을 호출합니다. 비어 있으면 → XDP_ABORTED를 반환하여 루프를 종료합니다.
08
XDP_GEN이 패킷 프레임 채움. pop한 메타데이터를 사용하여 사전 할당된 xdp_frame에 ACK 패킷 헤더(Ethernet, IP, TCP)를 구성합니다. XDP_TX를 반환합니다.
09
커널이 ACK 전송. XDP_TX 액션이 dev_xdp_enqueue()를 통해 생성된 ACK 패킷을 NIC 전송 큐로 보냅니다.
10
루프 반복 또는 종료. 큐에 대기 중인 각 ACK에 대해 07–09 단계가 반복됩니다. XDP_ABORTED 또는 budget 소진 시, 나머지 프레임은 재활용되고 dev_xdp_flush()가 완료됩니다.
핵심 성능 인사이트: ACK 병합(coalescing)이 자연스럽게 발생합니다 — 하나의 NAPI poll에서 처리된 여러 데이터 패킷이 여러 ACK 메타데이터 항목을 생성하고, XDP_GEN이 이를 마지막에 ACK 패킷으로 배치 처리합니다. 이는 Google Falcon, TAS, Caladan, eRPC의 병합 메커니즘과 유사합니다.

시퀀스: Homa Credit 생성

Homa의 경우 흐름은 유사하지만 XDP_GEN 프로그램이 더 복잡합니다:

H1
XDP_GEN이 credit 리스트에서 RPC 선택. bpf_rbtree_lower_bound()를 사용하여 우선순위(남은 바이트)별로 후보 RPC를 찾습니다. peer당 최대 1개의 RPC만 선택되도록 보장합니다.
H2
RPC 상태 조회. eBPF hashmap에서 RPC 상태를 쿼리하고, 남은 바이트와 우선순위에 따라 credit을 할당합니다.
H3
완료된 RPC 제거. credit 리스트(bpf_rbtree)에서 완료된 RPC를 정리합니다.
H4
credit 패킷 전송. 사전 할당된 프레임에 credit 패킷 데이터를 채우고 XDP_TX를 반환합니다. 복잡한 로직을 여러 프로그램으로 분할하기 위해 bpf_tail_call을 사용하여 verifier의 명령어 수 제한을 만족시킵니다.

데이터 구조 상세

per-CPU ACK 메타데이터 큐

XDP(수신)와 XDP_GEN(패킷 생성) 사이의 다리 역할. XDP가 push하고, XDP_GEN이 pop합니다.

/* eBPF 프로그램 측 — ACK 메타데이터용 per-CPU 큐 */ /* 새로운 타입이 아닌 표준 eBPF map 사용 */ struct { __uint(type, BPF_MAP_TYPE_QUEUE); /* 또는 PERCPU 변형 */ __uint(max_entries, 256); __type(value, struct ack_meta); } ack_queue SEC(".maps"); /* TCP ACK 메타데이터 구조체 (★ 전송 프로토콜별 정의) */ struct ack_meta { __u32 saddr, daddr; __u16 sport, dport; __u32 seq; __u32 ack_seq; __u16 window; /* 전송 프로토콜에 따라 추가 필드 가능 */ }; /* XDP 프로그램 내 (수신 경로): */ bpf_map_push_elem(&ack_queue, &meta, BPF_EXIST); /* XDP_GEN 프로그램 내 (패킷 생성): */ int ret = bpf_map_pop_elem(&ack_queue, &meta); if (ret < 0) return XDP_ABORTED; /* 큐 비어 있음, 생성 중단 */

PKT_QUEUE (BPF_MAP_TYPE_PKT_QUEUE) ★

페이싱을 위한 새로운 eBPF map 타입. xdp_frame 포인터를 저장하는 슬롯이 있는 버킷들을 포함합니다.

/* ★ eTran이 추가한 새 eBPF map 타입 (kernel/bpf/) */ /* 논문 §3.2.2에서 재구성 */ struct pkt_queue_bucket { struct xdp_frame *slots[BUCKET_SIZE]; int head, tail; /* 타이밍 휠용: 시간 슬롯에 매핑 */ /* credit 기반용: RPC ID에 매핑 */ }; struct bpf_map_pkt_queue { struct bpf_map map; /* 기본 eBPF map */ struct pkt_queue_bucket *buckets; __u32 nr_buckets; /* 타이밍 휠 상태 (속도 기반 페이싱용) */ __u64 current_tick; __u64 tick_interval_ns; }; /* 페이싱 메커니즘에 따른 사용 방식: */ /* * 속도 기반 (TCP/DCTCP): bucket_idx = tx_timestamp / tick_interval * → Carousel 스타일 타이밍 휠 구현 * * Credit 기반 (Homa): bucket_idx = RPC_ID * → RPC별 패킷 큐잉, credit 보충 시 디큐 */

확장된 xdp_md 컨텍스트

/* include/uapi/linux/bpf.h — eTran 확장 */ struct xdp_md { __u32 data; __u32 data_end; __u32 data_meta; __u32 ingress_ifindex; __u32 rx_queue_index; __u32 egress_ifindex; /* v6.6 기존 필드 */ /* ★ eTran 추가: UMEM 소유권 식별 */ __u32 umem_id; /* XDP_EGRESS용: 패킷이 해당 앱에 속하는지 검증 */ /* XDP (수신)용: 무효/접근 불가 */ /* XDP_GEN용: 해당 없음 (커널 생성 프레임) */ };

eBPF 프로그램 측 (전송 프로토콜 개발자 관점)

XDP_GEN 프로그램 구현 시 전송 프로토콜 개발자가 작성하는 내용을 보여줍니다. §4에서 기술된 TCP 및 Homa 구현을 기반으로 합니다.

TCP ACK 생성기 (XDP_GEN 프로그램)

SEC("xdp_gen") int tcp_ack_gen(struct xdp_md *ctx) { struct ack_meta meta; int ret; /* 단계 1: per-CPU 큐에서 ACK 메타데이터 pop */ ret = bpf_map_pop_elem(&ack_queue, &meta); if (ret < 0) return XDP_ABORTED; /* 더 이상 보낼 ACK 없음 */ /* 단계 2: 사전 할당된 패킷 프레임 접근 */ void *data = (void *)(long)ctx->data; void *data_end = (void *)(long)ctx->data_end; /* 단계 3: Ethernet 헤더 구성 */ struct ethhdr *eth = data; if ((void *)(eth + 1) > data_end) return XDP_DROP; /* bpf_fib_lookup 또는 캐시된 결과로 MAC 주소 채움 */ /* 단계 4: IP 헤더 구성 */ struct iphdr *ip = (void *)(eth + 1); if ((void *)(ip + 1) > data_end) return XDP_DROP; ip->saddr = meta.saddr; ip->daddr = meta.daddr; ip->protocol = IPPROTO_TCP; ip->tot_len = bpf_htons(sizeof(*ip) + sizeof(struct tcphdr)); /* ... 나머지 IP 필드 채움 ... */ /* 단계 5: TCP ACK 헤더 구성 */ struct tcphdr *tcp = (void *)(ip + 1); if ((void *)(tcp + 1) > data_end) return XDP_DROP; tcp->source = meta.sport; tcp->dest = meta.dport; tcp->seq = bpf_htonl(meta.seq); tcp->ack_seq = bpf_htonl(meta.ack_seq); tcp->ack = 1; tcp->window = bpf_htons(meta.window); /* ... 체크섬 계산 ... */ /* 단계 6: ACK 패킷 전송 */ return XDP_TX; }

Homa Credit 생성기 (XDP_GEN 프로그램, tail call 사용)

/* Homa는 복잡한 로직을 분할하기 위해 bpf_tail_call 사용 (논문 §4.2) */ /* 상태는 .bss 섹션의 per-CPU 변수를 통해 전달 */ struct { __uint(type, BPF_MAP_TYPE_PROG_ARRAY); __uint(max_entries, 4); __type(key, __u32); __type(value, __u32); } gen_progs SEC(".maps"); /* tail call 프로그램 간 상태 전달용 per-CPU 상태 */ struct gen_state { __u64 last_min_bytes; __u32 last_peerid; __u32 credits_allocated; } gen_ctx SEC(".bss"); SEC("xdp_gen") int homa_credit_gen_phase1(struct xdp_md *ctx) { /* 1단계: bpf_rbtree credit 리스트에서 후보 RPC 선택 */ /* bpf_rbtree_lower_bound()로 우선순위별 RPC 탐색 */ /* peer당 최대 1개 RPC 보장 (2-level rbtree 에뮬레이션) */ /* 선택 결과를 gen_ctx (.bss per-CPU)에 저장 */ bpf_tail_call(ctx, &gen_progs, 1); /* → 2단계로 */ return XDP_ABORTED; } SEC("xdp_gen/phase2") int homa_credit_gen_phase2(struct xdp_md *ctx) { /* 2단계: RPC 상태 조회, credit 할당 */ /* 3단계: 완료된 RPC를 credit 리스트에서 제거 */ bpf_tail_call(ctx, &gen_progs, 2); /* → 3단계로 */ return XDP_ABORTED; } SEC("xdp_gen/phase3") int homa_credit_gen_phase3(struct xdp_md *ctx) { /* 4단계: 선택된 RPC에 대한 credit 패킷 구성 및 전송 */ /* Ethernet/IP/Homa 헤더 채움 */ /* 전송 시 XDP_TX, 완료 시 XDP_ABORTED 반환 */ return XDP_TX; }
